SPEC — Node Keys, Pairing, and Split Seals
April 3, 2026. Technical specification.
Status: Designed. Not implemented.
Abstract
Phantom needs three mechanisms for identity and trust:
Node key — a portable identity that survives device loss
One-time pairing code — a temporary key for secure node-to-node connection
Split seal — a cryptographic proof that two nodes were physically present together
Together, these form the foundation of the trust layer described in DELIBERATION_TRUST.md. The node key is who you are. The pairing code is how you connect safely. The split seal is proof that the connection happened in the physical world.
1. Node Key — Portable Identity
Problem
Today, node identity lives in IndexedDB. If the user clears browser data, changes phone, or reinstalls — their identity is gone. Every seal they made is orphaned. Their trust chain is broken.
Solution
The node identity is an Ed25519 keypair, encoded as a human-readable string that the user can export, store, and import on any device.
Format
Secret key (keep private — this IS your node):
  nphantom1secret...  (64 characters, base58)

Public key (share freely — this identifies your node):
  nphantom1public...  (44 characters, base58)
The prefix nphantom1 identifies it as a Phantom key. Similar to Nostr's nsec/npub pattern, but distinct to avoid confusion.
Generation
On first use:
  1. Generate Ed25519 keypair using Web Crypto API
  2. Encode secret key as nphantom1secret...
  3. Encode public key as nphantom1public...
  4. Store in IndexedDB
  5. Derive human-readable node name from public key hash
     (e.g., "quiet-river-a3f2" — same as today)
  6. Prompt user: "Save your key. If you lose it, you lose your node."
Export / Import
Export:
  Settings → Export Key → shows nphantom1secret... string
  User copies it, writes it on paper, saves it anywhere they choose.

Import:
  Settings → Import Key → paste nphantom1secret... string
  Phantom derives the public key, restores the node name,
  and the user is the same node on the new device.
What the key signs
Every shared seal is signed with the node's secret key. Any node can verify the signature using the public key. This proves:
This seal was created by this specific node
The seal was not modified after signing
No one else could have created it (without the secret key)
What the key does NOT do
It does not identify a person. It identifies a node.
It does not prove the node is human. Only that the node is consistent.
It does not require registration, email, phone, or any external service.
Key compromise
If a secret key is stolen, the attacker can impersonate the node. There is no revocation mechanism today. The user must create a new key and re-establish trust from zero.
Future consideration: a revocation seal — a signed message that says "this key is dead" — published to the network. Nodes that see it stop trusting the old key. Not specified here.
2. One-Time Pairing Code — Secure Connection
Problem
When two nodes want to connect (via Bluetooth, local WiFi, or eventually relay), how do they know they're connecting to the right device? Without verification, a man-in-the-middle could intercept the connection.
Solution
A one-time code generated by one node, entered by the other. Valid for 60 seconds. Used once. Destroyed after use.
Flow
Node A wants to connect to Node B.

Node A:
  1. Generates a 6-digit code: 847291
  2. Displays it on screen
  3. Starts a 60-second countdown
  4. Listens for incoming connection with this code

Node B:
  1. Opens "Connect to node" 
  2. Enters the 6-digit code: 847291
  3. Sends connection request with code

Handshake:
  1. Node A receives request with matching code
  2. Both nodes perform Diffie-Hellman key exchange
  3. Shared secret established — all further communication encrypted
  4. Code is destroyed — cannot be reused
  5. Connection is live
Technical details
Code generation:
  code = first 6 digits of SHA-256(random 32 bytes + timestamp)
  Display as: XXX-XXX (e.g., 847-291) for readability

Key exchange:
  X25519 Diffie-Hellman via Web Crypto API
  Each node generates ephemeral X25519 keypair
  Exchange public keys over the initial connection
  Derive shared secret
  Encrypt all subsequent messages with AES-256-GCM using shared secret

Expiration:
  Code expires after 60 seconds OR after first use, whichever comes first
  Node A stops listening after expiration
  No retry with same code — generate new one
What this prevents
Accidental connection: you don't accidentally sync with a stranger's node
Replay attack: the code is single-use, useless after expiration
Eavesdropping: the Diffie-Hellman exchange creates a unique encryption key per session
What this does NOT prevent
Compromised device: if Node B's device is compromised, the attacker is in the room and has the code
Social engineering: someone who can read the code off your screen can connect
Both require physical presence — which is the point. The threat model assumes physical proximity is trusted.
3. Split Seal — Proof of Physical Presence
Problem
How do you prove that two nodes were physically together at a specific moment? A signed message proves a node exists digitally. Physical presence requires something more.
Solution
A seal that neither node can create alone. Each contributes half. The combined seal proves joint presence.
Flow
Node A and Node B are physically together.
They want to create a trust bond.

Step 1 — Each writes a fragment
  Node A writes: "The sky outside is gray"
  Node B writes: "and the coffee is getting cold"
  
  Neither sees the other's fragment yet.

Step 2 — Exchange fragments over encrypted channel
  (Using the pairing code connection from Section 2)
  Node A sends their fragment to Node B
  Node B sends their fragment to Node A
  Both now have both fragments

Step 3 — Combine and seal
  Combined idea: "The sky outside is gray | and the coffee is getting cold"
  The separator "|" is fixed by protocol
  
  Both nodes independently compute:
    moment = current timestamp (synced to nearest second)
    stamp = SHA-256(combined idea + moment)
  
  Both nodes verify they got the same stamp.
  If stamps match — the seal is valid.

Step 4 — Sign and store
  Node A signs the seal with their secret key
  Node B signs the seal with their secret key
  
  The split seal contains:
    - Combined idea
    - Moment
    - Stamp
    - Node A's public key + signature
    - Node B's public key + signature
    - Type: "presence"

Step 5 — Both nodes store the split seal
  This seal proves:
    ✓ Node A and Node B were present at the same moment
    ✓ Both contributed content (neither could create alone)
    ✓ Both signed it (both had their secret keys)
    ✓ The content was unique to this encounter (not replayable)
Wire format
{
  "type": "PH_SPLIT",
  "seal": {
    "i": "The sky outside is gray | and the coffee is getting cold",
    "m": "2026-04-03T15:30:00.000Z",
    "s": "a1b2c3...64 hex chars",
    "t": "presence",
    "fragments": [
      {
        "node": "nphantom1public_A...",
        "fragment": "The sky outside is gray",
        "signature": "sig_A..."
      },
      {
        "node": "nphantom1public_B...",
        "fragment": "and the coffee is getting cold",
        "signature": "sig_B..."
      }
    ]
  }
}
Verification by any third party
Anyone can verify a split seal:
1. Reconstruct combined idea from fragments: fragment_A + " | " + fragment_B
2. Verify stamp: SHA-256(combined idea + moment) == stamp
3. Verify signature A: Ed25519_verify(public_key_A, stamp, sig_A)
4. Verify signature B: Ed25519_verify(public_key_B, stamp, sig_B)
5. If all pass — these two nodes were together at that moment
Why this works
A bot can't create a split seal alone. It needs a second node to contribute a fragment. The bot doesn't know what the other fragment will be.
A replay is detectable. The moment is unique. Two split seals with the same fragments but different moments are two different encounters. Same fragments at the same moment from different locations is physically impossible.
The content is unpredictable. Each person writes something only they would write in that moment. "The sky outside is gray" is only true right now, at this place. It's a proof of context, not just a proof of key possession.
Both must consent. Neither can force the other to contribute. The seal requires active participation from both sides.
What this enables
Split seals are the building blocks of the trust network:
Trust chain example:

  Node A (genesis) ←split seal→ Node B
  Node B ←split seal→ Node C
  Node C ←split seal→ Node D

  Any node can trace:
    D → C → B → A (genesis)
  
  Each link is a verified physical encounter.
  The chain is as strong as its links.

How the three mechanisms work together
1. FIRST USE
   → Node generates keypair
   → User saves their nphantom1secret... key
   → Node is "unverified" — can seal privately, can share

2. MEETING ANOTHER NODE
   → Node A displays one-time code (847-291)
   → Node B enters code
   → Encrypted channel established
   → Both write their fragments
   → Split seal created and stored by both
   → Both are now "verified" to each other

3. SYNCING SEALS
   → Over the encrypted channel, both nodes sync shared seals
   → Each verifies incoming seals (SHA-256 + signature)
   → Feed grows with verified content

4. OVER THE RELAY (future)
   → Node connects to Phantom relay
   → Publishes shared seals (signed with node key)
   → Receives seals from other nodes
   → Seals from verified nodes (with split seal chain) are shown prominently
   → Seals from unverified nodes are shown as "new"

5. ON A NEW DEVICE
   → User imports nphantom1secret... key
   → Same node identity restored
   → Trust chain intact — split seals reference the public key, not the device
Implementation notes
Web Crypto API availability
All cryptographic operations use the Web Crypto API, available in all modern browsers:
Ed25519 — supported in Chrome 113+, Edge 113+, Safari 17+, Firefox 128+
X25519 — same support range
AES-256-GCM — universally supported
SHA-256 — universally supported
For older browsers, Ed25519 operations can fall back to a pure JavaScript implementation (TweetNaCl.js, ~6KB, no external dependency).
Storage
Node key stored in IndexedDB under the meta store:
{ "key": "nodeKey", "value": { "secret": "...", "public": "..." } }
Split seals stored in the seals store with type "presence":
{ "stamp": "abc...", "type": "presence", "fragments": [...] }
UI considerations
Key export: show the string with a COPY button. Warning: "Anyone with this key IS your node."
Key import: single text field. Paste and go.
Pairing code: large digits, centered, countdown timer visible.
Split seal creation: two text inputs side by side. "Write something only true right now."
Trust indicators: small icon next to node name in feed — unverified (no icon), verified (one dot), trusted (three dots), genesis (star).
Open questions
Key derivation from passphrase: Should users be able to generate a key from a memorable phrase instead of saving a random string? Easier to remember, but weaker if the phrase is guessable. Not specified here.
Multi-device sync: If a user imports their key on two devices, both are the same node. Seals created on device A don't automatically appear on device B. They need to sync (Bluetooth, relay, or manual export). This is expected behavior but may confuse users.
Split seal with more than two nodes: A group encounter could create an N-way split seal. Each person contributes a fragment. The combined seal proves all N were present. Not specified here — start with two.
Fragment content guidance: Should the protocol suggest what to write? "Describe something you both can see right now." Helps with uniqueness. But prescribing content feels un-Phantom.
Time sync between devices: Both nodes need approximately the same clock to generate matching timestamps. If clocks differ by more than 5 seconds, the stamps won't match. Mitigation: one node proposes the timestamp, the other accepts. Both compute with the agreed timestamp.
Category theory note
The split seal is a pullback in categorical terms. Two morphisms (fragments from A and B) converge on a shared object (the combined seal). The pullback is the unique object (the split seal) through which both morphisms factor. It exists if and only if both sources provide their part.
The node key makes each node a well-defined object in the category. The split seal is the morphism that connects them. Without keys, there are no objects. Without split seals, there are no arrows. Both are necessary for the category to have structure.
